## 5. Risks observed

- **R-1. No project references (`references: []`) yet.** Each package is `composite: true` (per TASKS §3.2), but the root and consumer apps do not declare `references`. `tsc --build` from the root is therefore not wired. Downstream specs (especially SPEC-0005 when shared contracts start to be consumed) should add references so that incremental builds work across packages. Today, `pnpm -r build` succeeds because each project compiles independently against the empty placeholders.
- **R-2. `moduleResolution: Bundler` is friendly for Vite/Web but mismatches what Node 22 actually uses at runtime (`NodeNext`).** Apps that ship Node code (`apps/api`, `apps/sri-core`) may need per-app overrides when they start importing third-party packages with subpath/conditional exports. This will be discovered the first time a real dep is added.
- **R-3. No `engines.node` enforcement at `pnpm install`.** `pnpm` will warn but not fail by default. CI should be configured to fail on Node-version mismatch in a later spec.
- **R-4. `pnpm-lock.yaml` is committed conceptually but not by this prompt.** The lockfile exists in the working tree but is untracked. Whoever creates the first commit must include it.
- **R-5. `.tsbuildinfo` files appear under `packages/*/` (from `composite: true`).** Already covered by `.gitignore` (`*.tsbuildinfo`). Verified via `git check-ignore`.
- **R-6. No `.gitattributes`** — line-ending normalisation relies on `.editorconfig` alone, which is editor-side, not git-side. SPEC-0001 §6.9 has the canonical content; recommend a follow-up spec adds it.
- **R-7. The shipped `tsconfig.base.json` cannot be exercised via `tsc --showConfig` directly** (see D-7). This is a TS limitation for pure base configs and is _only_ a documentation/validation friction, not a functional defect.

- **F-8 (Empty placeholder content)**: every `src/index.ts` is a `placeholder` export. Each downstream spec (SPEC-0005, SPEC-0006, SPEC-0010, …) must replace these with real code.

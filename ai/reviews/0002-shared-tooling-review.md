# REVIEW-0002 — Shared tooling (ESLint flat config, Prettier, Lefthook, commitlint, CI)

## 1. Summary

Wired the project-wide developer-experience layer on top of the PROMPT-0001 monorepo
skeleton. A single ESLint 9 flat config now lives in `@facturador/config/eslint` and
is re-exported by the root plus every workspace member. Prettier 3, EditorConfig and
a root `.prettierignore` keep formatting deterministic. Lefthook is installed with a
`pre-commit` hook (eslint --fix + prettier --write on staged files) and a
`commit-msg` hook backed by commitlint + `@commitlint/config-conventional`. A
GitHub Actions workflow (`.github/workflows/ci.yml`) defines three jobs — `lint`,
`typecheck`, `test` — on `ubuntu-latest` with Node 22 and pnpm 9. All six
finishing-line validations (`pnpm install`, `pnpm -r lint`, `pnpm -r format:check`,
`pnpm -r test`, `pnpm -r typecheck`, `pnpm -r build`) exit 0 from a clean tree.
Forced-failure validations §5.1–5.4 each fired the expected ESLint or commit-msg
rule, then were reverted to a clean state.

The locked-down security rules — `no-console: error`, `no-restricted-syntax` for
`process.env`, and `no-restricted-imports` for cross-workspace relative paths —
were each exercised against a deliberate violation; every one fails with exit
code 1 and surfaces the project's diagnostic message.

## 5. Risks observed

- **R-1. `typescript-eslint` strict-type-checked rules are expensive.** Each
  `pnpm exec eslint .` walk takes ~3–4 s on the empty tree. Once SPEC-0005 onward
  adds real code, expect 10–20 s. Mitigations: per-package incremental cache (`--cache`)
  in a later spec; CI matrix caching of the `.eslintcache`.
- **R-2. `eslint-plugin-import`'s TS resolver is wired via `settings["import/resolver"].typescript = true`** but no `eslint-import-resolver-typescript` is installed yet. Today no `import/...` resolution rule fires (default behaviour), so the lack of the resolver is silent. As soon as `import/no-unresolved` or `import/order` is enabled, the resolver must be added. Flagged as F-1.
- **R-3. The Lefthook `pre-commit` invokes `pnpm eslint --fix {staged_files}` and `pnpm prettier --write {staged_files}`.** On the first run after cloning, `pnpm eslint` resolves through the root devDep — fine. But if a contributor's pnpm store is empty, the hook's first invocation will be slow (≈3 s). Lefthook's `parallel: true` already mitigates this.
- **R-4. `vitest@2.1.4` is at root only.** Each workspace's `test` script calls `vitest run --passWithNoTests` but vitest itself resolves via pnpm's hoist to the root. If a workspace later needs a workspace-specific vitest, it must add its own devDep. Today this is acceptable for placeholders.
- **R-5. Lefthook's auto-generated `lefthook.yml`** was overwritten with the spec-mandated content. The first `pnpm install` after a clean clone re-runs `lefthook install` via the root `prepare` script — verified to succeed.
- **R-6. The forced-failure tests produced `@typescript-eslint/no-unsafe-*` errors _alongside_ the project-mandated rule** (because `console`, `process` and unresolved import names are typed as `error` in the strict ruleset). This is harmless — the project rule still fires and is named distinctly in the output — but if a future contributor disables `@typescript-eslint/no-unsafe-call` for some reason, the forced-failure surface area shrinks. The project rules themselves are still load-bearing.
- **R-7. No `eslint-plugin-security`** yet. Many "security" lints — `detect-eval-with-expression`, `detect-non-literal-require`, etc. — are nice-to-have defensive layers. Not blocking for SPEC-0002, but listed as a follow-up.

None of the file-level overrides re-enable `process.env` access outside the

anywhere. No `console` allowance leaks into production source. The `eslint-config-prettier`
preset is loaded last in the chain so it only disables _formatting_ rules, never
the security rules above.

Other security observations:

- `.prettierignore` excludes `pnpm-lock.yaml`, `*.tsbuildinfo`, `node_modules`, `dist`, `build`, `coverage`, `.turbo`, `.git`. No certificate/secret-bearing glob (e.g. `*.p12`) was added — they are already excluded by `.gitignore` from PROMPT-0001, and Prettier wouldn't format them anyway.
- The CI workflow uses `actions/checkout@v4`, `pnpm/action-setup@v4`, and `actions/setup-node@v4` — all major versions in current security support — and grants only `permissions.contents: read` (least-privilege). No deploy or publish steps were added.
- Lefthook's hooks do not interactively prompt and do not log secrets. They run `eslint --fix` and `prettier --write` only on `{staged_files}`, never the whole tree.
- The forced-failure test fixtures never embedded real credentials, RUCs, or sample certificate material; just synthetic placeholder identifiers (`FOO`, `whatever`).

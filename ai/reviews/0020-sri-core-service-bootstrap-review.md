# REVIEW-0020 — SRI Core service bootstrap

## 1. Summary

Stood up the public surface of `apps/sri-core` exactly as PROMPT-0020 demands:

- New Prisma models for SRI persistence (`Certificate`, `SriDocument`, `SriEvent`, `BurnedSecuencial`).
- HS256 service-JWT round-trip between `apps/api` and `apps/sri-core`,
  with the canonical claims contract (`iss=api`, `aud=sri-core`, `sub=companyId`,

  `POST /v1/documents/:claveAcceso/resend`. Plus `/healthz` and `/readyz`

  `SRI_STUB_MODE=true` with a `stub_mode_in_production` message; a Vitest
  child-process test spawns the loader and asserts non-zero exit.
- Stub mode walks the document through PENDIENTE → FIRMADO → ENVIADO →
  RECIBIDA → AUTORIZADO via the legal state machine — no shortcuts.

## 4. JWT claims contract

| Field | Value                                                                       |
| ----- | --------------------------------------------------------------------------- |
| `alg` | `HS256` (HMAC-SHA-256). Pinned on both sides.                               |
| `typ` | `JWT`.                                                                      |
| `iss` | `api`.                                                                      |
| `aud` | `sri-core`.                                                                 |
| `sub` | ULID `companyId` of the tenant making the call.                             |
| `iat` | epoch seconds at mint time.                                                 |
| `exp` | `iat + 60` (hard cap; `mintServiceJwt` rejects higher TTLs).                |
| `jti` | Fresh ULID per call. Not yet a replay deny-list — logged for observability. |

Verifier policy:

- `algorithms: ["HS256"]` — rejects `alg:none` and `alg:RS256` (alg
  confusion) before signature work.
- `issuer: "api"`, `audience: "sri-core"` — claim validation by `jose`.
- `clockTolerance: 5` (seconds) — soaks small drift; long-past tokens
  still rejected.
- `sub` must be a non-empty string; absence yields `missing_subject`.
- All failure reasons collapse to a single 401 `sri.service_token_invalid`
  on the wire. Structured reason is logged server-side via `req.log.warn`
  (never logged at the JWT level — REDACT_PATHS covers `Authorization`).

## 7. State machine matrix (as implemented)

Per SPEC-0026 §6.2 + `apps/sri-core/src/lifecycle/transitions.ts`. Each row
is the `from` state; each cell is the set of legal `to` states.

| ----------------- | :--: | :-: | :-: | :---: | :--: | :--: | :-----: | :-: | :-----: | :-----: |
| **PENDIENTE**     |      |  ✓  |  ✗  |       |      |      |         |     |    ✓    |    ✓    |

| **AUTORIZADO**    |      |     |     |       |      |      |         |     |         |         |
| **NO_AUTORIZADO** |      |     |     |       |      |      |         |     |         |         |

Notes:

- "self" = the matrix allows the self-loop because retry-bookkeeping
  needs to log a fresh event. `recordEvent` only honours it when called
  with `allowSelfLoop: true`; without the flag, same-state writes throw
  `sri.invalid_transition`.
- Terminal states (`AUTORIZADO`, `NO_AUTORIZADO`, `DEVUELTA`,
  `ERROR_BUILD`) have empty outgoing rows.
- The skeleton allows `PENDIENTE → ERROR_RED` to give the orchestrator
  a path to record a network failure before signing.

## 8. Security review (PROMPT-0020 §6)

1. **`SERVICE_JWT_SECRET` loading.** Read only by `apps/sri-core/src/env.ts`
   and `apps/api/src/env.ts`. ESLint's `no-restricted-syntax` rule keeps
   `process.env` out of every other file (per-file override only on
   `src/env.ts`). REDACT_PATHS (`packages/logger/src/redactions.ts`) already
   includes `SERVICE_JWT_SECRET` + `*.SERVICE_JWT_SECRET`.
2. **TTL ≤ 60 s.** Enforced by `mintServiceJwt` (rejects out-of-range)
   and re-validated by `verifyServiceJwt` (rejects expired beyond 5 s
   skew). Integration test asserts a > 1 hr-old token is rejected.
3. **No PII in claims.** Only `iss/aud/sub/iat/exp/jti`. `sub` is a
   26-char ULID with no identifying entropy.
4. **Cert columns never logged.** REDACT_PATHS already covers `p12*`,
   `*.privateKey`, `certificatePassphrase`, `signedXml`, etc. The
   schema's `p12CiphertextB64`, `passphraseCiphertextB64`, etc. names
   match the existing fields covered by the redaction list (and the
   `p12*` wildcard handles forward-compatible additions).
5. **Audit logging.** Emit handler writes a `req.log.info` line with
   `event: "sri.emit.persisted"` carrying only `companyId`,
   `documentId`, `estado`. No tokens, no claveAcceso in the message,
   no XML.
6. **Stub mode in production refused.** Env loader's `superRefine`
   throws on `NODE_ENV=production && SRI_STUB_MODE=true`. The
   `refuse-to-boot.test.ts` spawns the process via `tsx src/index.ts`
   and asserts exit code != 0 plus `stub_mode_in_production` on stderr.
7. **Web app never calls sri-core.** Only `apps/api/src/sri/client.ts`
   talks to sri-core, and only via the `mintServiceJwt` + `sriCoreFetch`
   pair. Web → API is the only path from the browser.

## 9. Deviations from spec/plan

1. **JWT library.** SPEC-0020 §6.3 illustrated the verifier with
   `jsonwebtoken`. We chose `jose@5.9.6` instead: it's fully ESM (no
   `Buffer` polyfill on the web side later), and it rejects `alg: none`
   by default. The `algorithms` filter still pins HS256 explicitly. The
   `mintServiceJwt` shape matches the spec.
2. **Resend handler.** The full resend pipeline lives in SPEC-0026's
   orchestrator. This slice returns `422 validation.failed` (informativo
   tipo) for resend requests on non-terminal documents. Terminal states
   correctly throw `409 sri.invalid_transition`. PROMPT-0026 will
   replace the placeholder.
3. **Error code casing.** SPEC-0006 ProblemDetail `code` requires
   `^[a-z][a-z0-9_]*(\.[a-z][a-z0-9_]*)*$`. The natural
   `NotFoundError("sriDocument")` would have produced `sriDocument.not_found`,
   which fails the regex and falls back to `internal.unexpected`. We
   pass `sri_document` instead, so the wire emits `sri_document.not_found`.
4. **`scripts/` workspace.** Added a fourth workspace root (`scripts`) so
   the helper script can `import { SignJWT } from "jose"` cleanly. The
   script duplicates ~25 lines of mint logic so it doesn't depend on the
   `@facturador/utils` build artefact (keeps the script runnable even if
   `dist/` is dirty).
5. **`Certificate` model.** The PLAN-0020 sketch was extended slightly to
   reserve `serialNumber` and `passphraseCiphertextB64/Nonce/Tag` columns.
   SPEC-0021 will populate those; reserving here avoids a second
   migration. `(companyId, serialNumber)` is unique to prevent re-upload
   collisions per SPEC-0021 §6.4.
6. **`BurnedSecuencial` skeleton.** Minimal table per the PROMPT hard
   constraint ("BurnedSecuencial for SPEC-0030; minimal skeleton OK
   here"). SPEC-0030 will expand it.
7. **`/health` retained.** The original `/health` endpoint stays as an
   alias for `/healthz` so the existing docker-compose healthcheck +
   `error-model.test.ts` snapshots don't regress. SPEC-0020 §FR-7
   nominates `/healthz`; both work.
8. **api Dockerfile not touched.** The pre-existing `apps/api/Dockerfile`
   does not copy `packages/db` — a regression that pre-dates this
   slice. Out of scope; recorded as a follow-up.

## 10. Risks observed

1. **Shared-secret rotation.** A single `SERVICE_JWT_SECRET` powers both
   sides. Rotating requires a brief downtime or a two-step deploy (mint
   with new, verify with both). SPEC-0020 §13 already nominates RS256
   as a future option; we keep the open question intact.
2. **`jti` not enforced for replay defence.** Tokens carry a `jti` but
   verify-side does not consult a deny-list. Within the 60 s window an
   attacker who intercepted a token (TLS termination compromise) could
   replay it. Mitigation: short TTL; future deny-list via Redis or
   Postgres advisory locks.
3. **No rate limit on `/v1/documents/*`.** api can hammer sri-core
   without throttling. The api already rate-limits the user-facing
   login route; an internal token bucket here would belong in
   SPEC-0033's orchestrator.
4. **Single Prisma client across services.** Both api and sri-core
   route their writes through the same `@facturador/db` singleton.
   Cross-service drift (api accidentally touching `SriDocument`) is
   prevented today by code review only; an ESLint `no-restricted-imports`
   rule excluding api from `@prisma/client.sriDocument` could harden
   this.
